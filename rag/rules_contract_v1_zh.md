# 规则契约 v1（可检查）

## 目的
将输出风格约束从"临时聊天偏好"升级为"仓库内可执行规则"。

## Rule IDs

### R-ASK-001
- 名称：提问题禁止总结与比喻
- 要求：提问时只允许"问题位置 + 问题本身"。
- 禁止：先做总结、引入类比/隐喻、先给宏观概述再提问。
- 合格示例：
  - 位置：第 3 段第 2 句；问题：这里将"找对人"和"分好钱"并列，但未给出衡量标准。

### R-ASK-002
- 名称：一次对外提问上限 = 3
- 要求：一次回复中 `请你拍板的方向` 段条数 `<= 3`。
- 处理：超出时必须压缩、合并，或把可假设项降级到 `默认假设` 段。
- 例外：有 P0 阻塞时，P0 先问，P1 顺延到下次攒齐再一次性提。

### R-SUM-001
- 名称：总结禁止比喻与比喻式改写
- 要求：总结直接陈述事实与判断。
- 禁止：使用比喻、借物喻人、用"换一种说法"进行比喻式重写。

### R-CODE-001
- 名称：代码修改请求只输出完整可执行命令
- 要求：涉及代码改动时，先内部形成思路，最终仅输出完整命令（可多条）。
- 禁止：输出完整代码块作为最终交付。

### R-LANG-001
- 名称：L1 输出必须使用中文标签
- 要求：必须使用新 6 段中文标签 —— `当前判断` / `假定分支` / `请你拍板的方向` / `默认假设` / `证据` / `你的下一步`。
- 禁止：使用 `Now` / `Next` / `Branches` / `Blockers` / `Blocker` / `Assumptions` / `Assumption` / `Evidence` / `Self-check`，以及这些词的任何中英混写变体（如 `Blockers（P0/P1）` / `## 请你拍板的方向（Decisions）`）。
- 例外：仅在用户一次性显式说"用英文输出 / English output"时才允许切换英文标签；下一次对话自动回落中文默认。
- 兼容：旧段名 `下一步动作` / `不确定的问题` 仍允许在历史 case 文件中存在，但新输出必须用新段名。

### R-FMT-001
- 名称：L1 输出禁止 meta 标题
- 要求：L1 输出第一行必须是正文第一个中文段标题（即 `## 当前判断`）。
- 禁止：在顶部加任何 meta 框架标题，包括但不限于 `L1 决策卡 — <case 名>` / `L1 xxx` / `L2 xxx` / `L3 xxx` / `<任意>决策卡` / `决策 card` / `轻量卡片` / `分析卡`。
- 原因：这些词是对"答题"这件事的比喻式包装，既违反 R-SUM-001（禁比喻），也暴露了内部分层标签（L1/L2/L3）给审阅者。
- 替代：若确需题目回显，放在 `当前判断` 首句以事实陈述方式说明，而不是作为独立标题行。

### R-FMT-002
- 名称：假定分支语法
- 要求：`假定分支` 段每条必须用 `假定情况 N（<触发条件描述>）：<对应方案动作>` 句式。
- 禁止：使用 `if ... then ...` 触发器语法（这是给系统/同事写的执行触发，不是给思考者读的分支）。
- 原因：考试场景下用户在"思考分支"而非"执行触发器"。`假定情况 N` 让分歧点清晰，更容易快速选定一条推进。
- 合格示例：`- 假定情况 1（部件问题为安全/合规风险）：立刻启动延期 + 替代供应商认证 + 透明沟通`
- 不合格示例：`- if 部件为安全风险 then 启动延期`（违反，重写为假定情况句式）

### R-RESEARCH-001
- 名称：拍板项必须先双轨研究后陈列
- 要求：在 L1 输出 `请你拍板的方向` 段之前，模型必须**在内部**对每个候选拍板项跑**双轨研究**——
  - **轨道 A：业界 best practice**。研究路径按成本由低到高：
    1. 先扫 `skills/` 目录看有无现成可复用 SOP（如 HTML 制作、可视化范式、常见算法/库选型）
    2. 查 `rag/anker_values_rag_v1_zh.md` 是否对该决策点有指引
    3. 扫题目原文找暗示词（"教学"/"评估"/"对比"/"科普" 等）
    4. 必要时调用 web search 拉外部 best practice 引用
  - **轨道 B：第一性原理切入**。回到题目本质问"若不参考任何既有做法，从公理 + 约束推出来的最优解是什么"。常见落点：经济成本结构 / 物理可行边界 / 数据定义层 / 用户根本需求。
  - **轨道 C：综合评审**。把 A 与 B 的结论摆在一起，输出"业界做法 X / 第一性原理 Y / 在本题情境下推荐 Z 因为 ⟨结合两轨的取舍依据⟩"。
- 处理：
  - 若 A 与 B 高度一致 + 无歧义：**降级到 `默认假设` 段**，标 `[best practice + 第一性原理一致结论]`，不上拍板。理由：用户不需要拍板两轨都已收敛的事。
  - 若 A 与 B 分歧 / 仍需用户判断：上 `请你拍板的方向`，每个选项必须呈现"业界做法 X / 第一性原理 Y" 两轨结论，让用户看到取舍依据再选。
  - **禁止把 best practice 和第一性原理简单二选一**：实际题目里两条都是可参考的，模型不能默认偏向某一条而隐藏另一条。
- 禁止：
  - 没做这两轨研究就把决策点直接抛给用户拍板
  - 在轨道 A 里跳过 `skills/` 目录扫描（这是低成本高价值的复用点）
- 例 1（两轨一致 → 降级）：题目"做一个对比 HTML"，"音频素材来源"——业界标准是"预置 2-3 段示例 + base64 内嵌"，第一性原理也支持（零依赖单文件最低传递摩擦），两轨一致 → 进 `默认假设`。
- 例 2（两轨分歧 → 上拍板）：题目"做一个 AI 客服方案"，"是否走 RAG 架构"——业界 best practice 倾向 RAG（成熟、可控），第一性原理可能倾向"端到端模型直接回答"（数据闭环更紧），两轨分歧 → 上拍板，每个选项呈现两轨结论。

### R-CLARITY-001
- 名称：拍板选项必须直接解释 + 必带引用 + union 不鼓励
- 要求：
  1. **术语必须直接解释 + 标引用来源**：选项内出现领域术语（如"齿音"、"共振峰"、"真伪指纹"、"RAG 架构"）时，必须用"X 指 Y"的直接陈述句解释，**并附引用来源**（标 `[依据：…]` 或 `[S0x]`），避免编造定义。引用形式可为权威书目、行业标准、本仓库 RAG 条目、或 web search 拉到的外部源。
  2. **选项陈述只用直接说明**：禁止比喻 / 借物喻人 / "换一种说法"式重写（这是对 R-SUM-001 在选项层的复用）。重复内容时直接重复，不要为避免重复而比喻。
  3. **union 选项不鼓励但不禁止**：若 A/B/C 都成立且确有"全做"路径，可以保留 D 但**必须明确标注** `D: 前 N 项叠加 = A + B + C`，并在 `我的推荐` 里说明"全做"的代价（成本 / 时间 / 优先级）。优先做法是把"全做"作为 `你怎么回复` 里的组合方式提示（如：`说"组合 A+B" 让我合并`），而不是占用一个选项位。
  4. **元信息禁污染选项陈述**：选项里禁出现"工作量 ×3" / "成本太高"这类只供模型自己评估的元信息，应挪到 `我的推荐` / `理由` 块说明。
- 合格示例（术语直接解释 + 带引用）：`C. 真伪指纹页 — "真伪指纹"指通过共振峰位置 / 基频抖动 / 频谱滚降等声学特征判断音频是真人录制还是 AI 合成 [依据：Praat 语音分析手册 §6.2]`
- 不合格示例（术语无解释 + 无引用）：`C. 真伪指纹页`（"真伪指纹"是什么？读者无法判断；模型可能编造）
- 不合格示例（用比喻重写）：`A. 教学科普页 — 像 Khan Academy 那种风格的科普页`（违反禁比喻；改为"给非专业读者直接解释概念，配前后对比"）
- 可接受的 union（带 N 项叠加标注）：`D: 前 3 项叠加 = A + B + C（多 Tab 形式）` + 在推荐里写"工作量约为单选项的 2.5 倍"

### R-EVID-001
- 名称：证据引用必须带锚点
- 要求：每条 `[Sxx]` 引用必须显式指出在支撑哪个段的哪个论点，格式：`[Sxx]（支撑 <段名> 中 "<具体论点>"）：<引用内容>`
- 禁止：漂浮引用，如 `[S01] Anker 价值观主线：用户第一 > 短期业绩。`（没说在支撑什么）
- 原因：审阅者需要立即知道这条证据为什么出现、它在保护哪一个判断。漂浮引用等于没引。
- 合格示例：`[S01]（支撑 当前判断 中 "用户第一优先于发布时间"）：Anker 价值观主线 —— 长期主义优先于抢时间窗口。`

### R-CHECK-001
- 名称：规则自检（内部执行 + 默认隐藏）
- 要求：每次输出前，模型在内部按以下 10 条逐项检查（R-ASK-001 / R-ASK-002 / R-LANG-001 / R-FMT-001 / R-FMT-002 / R-RESEARCH-001 / R-CLARITY-001 / R-EVID-001 / R-SUM-001 / R-CODE-001）；任一不过，重写后再输出。
- 强制门禁：`scripts/smoke_check.sh` 在产物存盘后做结构化校验，是真实的把关层。
- 对外输出：**默认不输出**这 8 条 Yes/No 列表。理由：用户读了 L1 前半部分内容好坏就已经判断了答案质量，展示规则名单对答题质量评估无新增信息，反而增加噪音。
- 例外：仅当用户在当前消息里显式说出 `带规则自检` / `规则自检` / `debug` / `自检模式` 之一时，才在末尾附 8 条 Yes/No；下一次回复自动回落隐藏。

## 冲突处理
当用户临时要求与本契约冲突时，优先遵守本契约。
