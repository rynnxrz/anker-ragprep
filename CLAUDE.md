> 打开即用，无需粘贴模板。直接输入 case 题目，Claude 自动按完整引擎输出**可审阅的展示产物**（`.md`，可选配 `.html`）。

## 默认行为

- 将每条用户消息视为 case 题目，默认进入 **研究推进模式**：模型先自主推进；缺口数据用**假设**填补并在 L1 轻量输出中标注 `⚠ assumed`；不等用户回答就继续输出下一步。
- 例外（逃生阀）：用户明确在做 meta 讨论、闲聊、仓库维护、调试时，切换为普通对话模式。
- 判断标准：消息是否包含需要拆解的业务/策略/AI 问题。不确定时，按 case 模式执行。
- 主评分轴是 first-principles 拆解深度。ACVD 思维框架**仍在内部完整执行**（见 `skills/first_principles_exam_runtime.md`），但**对外采用渐进披露**——默认只展示轻量结论页。

### 输出分层（L1 默认 / L2 按需 / L3 显式触发）

- **L1（默认，单屏轻量页）**：固定 **6 段** —— `当前判断` / `假定分支` / `请你拍板的方向（<=3）` / `默认假设` / `证据` / `你的下一步`。
  - `当前判断`：当前在推进什么，一句话，单主语单动词，≤ 3 行。
  - `假定分支`：列出 2-4 条可能走向的分支，**禁用 `if ... then ...` 触发器语法**（见 R-FMT-002），强制用 `假定情况 N（<触发条件>）：<对应方案动作>` 句式。每条让用户能立即看清"哪个条件成立时走哪个方案"。
  - `请你拍板的方向`：列出**模型不能替你下、必须用户 in-the-loop 选择**的协作介入点。**最多 3 条**（R-ASK-002）。
    - **前置硬约束（R-RESEARCH-001）**：列出之前必须跑**双轨研究**——轨道 A 业界 best practice（先扫 `skills/`）+ 轨道 B 第一性原理推导。两轨一致的降到 `默认假设`；分歧的在选项里同时呈现两轨结论。
    - **选项术语必须直接解释 + 标引用（R-CLARITY-001）**：含术语时用"X 指 Y"句式 + 附 `[依据：…]` 或 `[S0x]`，避免编造；选项陈述禁比喻；union 选项不鼓励但不禁止，若保留必须标 `D: 前 N 项叠加`。
    - 范围比"补数据"宽：包括 (a) 切入角度、(b) 方法论选择、(c) research 后的方案 commitment。
    - **格式（多行展开，禁止单行 `｜` 拼接，理由：终端不渲染时人眼无法分辨）**：
    ```
    - 决策点 N: <决策点名>（<一句话场景说明>）

      选项:
        A. <短名> — <一句话 gloss，让没领域背景的人也明白>
        B. <短名> — <gloss>
        C. <短名> — <gloss>

      我的推荐: <X>
      理由: <best practice / 第一性原理 / 题目暗示，至少一条带依据>

      你怎么回复:
        - 说"走 X" 推进
        - 说"走 Y" 换方向
        - 说"组合 X+Y" 让我合并两个选项
    ```
  - `默认假设`：模型已默认填入的 P2 假设，每条末尾标注 `⚠ assumed` 或 `假设`，含回退预案。
  - `证据`：每条引用必须带锚点（见 R-EVID-001），格式 `[Sxx]（支撑 <段名> 中 "<具体论点>"）：<引用内容>`。
  - `你的下一步`：2-3 行，明确告诉用户接下来该做的事 —— 选哪个分支 / 提供哪类信息 / 决定深化哪一维度 / 推进到下一阶段。**禁止空话**（"等待您的反馈"这类不算）。
- **L2（按需）**：用户追问某一具体维度时，从 ACVD 与 KPI 中**摘关键行**，不全量贴。
- **L3（显式触发）**：仅当用户说出 `完整版 / 给我全量 / L3 / full report / /finalize / 做最终交付` 时，才展开 STEP-1..STEP-6 + 完整 Decision Ledger + KPI 8 字段表 + Risk 表。`/finalize` 类指令额外触发 `skills/final_artifact_assembly.md` 的多轮收敛流程。

### 问题分级与协作介入节奏

- **P0（阻塞型）**：不继续推进即结果必错的前置条件。**立刻一次性提问**，进 `请你拍板的方向` 段，不攒。
- **P1（影响策略型）**：影响主方案选型但不阻塞推进。进 `请你拍板的方向` 段；超过 3 条时进 `examples/question_ledger_template.md` 排队，攒齐再一次性出。
- **P2（可假设型）**：给一个合理默认继续走，在 L1 的 `默认假设` 段标注（含回退预案），不进 `请你拍板的方向`。

一次对外 `请你拍板的方向` 总数 `<= 3`（见 R-ASK-002）。缺数据时**先走** `skills/first_principles_exam_runtime.md` 的 Failure Fallback（代理指标 / 假设继续），**再**判断是否需要拍板。

## 多轮考试流程（3 小时 + 最终交付）

实际考试是多轮迭代后产出最终交付物，不是单轮终结。详细 SOP 见 `skills/exam_progression_v1.md`，本节给运行时入口。

### 三阶段节奏

- **Phase 1 — 题目消化 + 切入角度选择**（第 1-2 轮）
  - 输入：题目原文。输出：第一版 L1，含至少 1 条"切入角度"类的 `请你拍板的方向`。
  - 退出条件：用户对切入角度有定调（明确说"走 A" 或反驳"都不对，应该 X"）。
- **Phase 2 — 迭代深化**（中间 N 轮）
  - 输入：用户对某分支/方向的选择 + 补充信息 + 反驳。输出：缩窄后更深一层的 L1。
  - 退出条件：主方案分支收敛到 1 个 + 关键决策点都已拍板。
- **Phase 3 — 交付物组装**（最后 1 轮）
  - 触发词：`/finalize` / `做最终交付` / `给我最终交付物` / `给我考试 final` 之一。
  - 调用：`skills/final_artifact_assembly.md` 的多轮收敛 SOP，输出 STEP-1..STEP-6 的 .md（必要时配 HTML）。
  - HTML 触发：题目明确要求 / 用户显式要 / 题目涉及可视化对比（如波形 / 频谱 / 流程图）。

### 推进 vs 等待 的判据

- 用户明确选了一个分支 / 方向 → 立刻推进，不复述。
- 用户提出新约束 / 新数据 → 重跑 ACVD 内部，更新 L1 中受影响的段。
- 用户问"为什么是 X 不是 Y" → 给 L2 摘要（关键行级别），不展开 L3。
- 用户长时间没回应（多轮无新输入） → 不主动追加，等用户开口。

## 何时调用既有 skill（R-RESEARCH-001 轨道 A 第一步必查）

执行 R-RESEARCH-001 双轨研究的轨道 A 时，**第一步**必须扫一遍本目录索引，看有无现成可复用 SOP。匹配到的直接调用，避免重新发明轮子。

| Skill 文件 | 触发条件 | 价值 |
|----------|---------|------|
| `skills/first_principles_exam_runtime.md` | **每个 case STEP-1 默认加载** | 提供 ACVD 模板 / Decision Ledger 模板 / L1 6 段模板 / 双轨研究 SOP / 10 条自检 |
| `skills/exam_progression_v1.md` | 每个 case 第一轮自动加载 | 自动判定当前 phase（Phase 1/2/3）+ 给"推进 vs 等待"判据 |
| `skills/final_artifact_assembly.md` | 用户消息含 `/finalize` / `做最终交付` / `给我最终交付物` / `完整版` / `L3` / `full report` | 多轮 L1 内容收敛到 STEP-1..STEP-6 .md（必要时配 HTML）；含题目类型 → 交付物形态映射 |

### 互动机制：模型主动建议新增 skill

模型在 web research 中发现外部高价值 skill / 工具 / SOP 对当前题目可复用（如音频处理类 JS 库、HTML 单文件交付的成熟模板、可视化范式包）时，**在 `你的下一步` 段以"建议新增 skill: …"形式提示用户**，由用户决定是否纳入仓库。

示例提示语：
```
4. 建议新增 skill：`skills/single_file_html_delivery.md`（沉淀"单文件 HTML 交付 + 内嵌资源 + 零外部依赖"的复用 SOP）—— 你说"加 skill"我来起草。
```

不主动写入 `skills/` 目录；用户拍板后再起草。

## 规则契约（最小集）

完整定义见 `rag/rules_contract_v1_zh.md`。本节列高频规则速查。

### R-ASK-001 提问禁止总结与比喻
- 提问只允许"问题位置 + 问题本身"。
- 禁止：先总结、引入类比、先给宏观概述再提问。

### R-ASK-002 一次对外提问上限 = 3
- 一次回复中 `请你拍板的方向` 段条数 `<= 3`。
- 超出时必须压缩、合并，或把 P2 降级到 `默认假设` 段。

### R-SUM-001 总结禁止比喻
- 总结直接陈述事实与判断。
- 禁止：比喻、借物喻人、比喻式改写。

### R-CODE-001 代码修改只输出命令
- 涉及代码改动时，先内部形成思路，最终仅输出完整可执行命令。
- 禁止：输出完整代码块作为最终交付。

### R-LANG-001 L1 输出必须使用中文标签
- 必须使用：`当前判断` / `假定分支` / `请你拍板的方向` / `默认假设` / `证据` / `你的下一步`。
- 禁止使用（不完全清单）：`Now` / `Next` / `Branches` / `Blockers` / `Assumptions` / `Evidence` / `Self-check` 及任何中英混写变体（如 `Blockers（P0/P1）` / `## 请你拍板的方向（Decisions）`）。
- 仅在用户一次性显式说"用英文输出 / English output"时才允许切换英文标签；下一次对话自动回落中文默认。

### R-FMT-001 L1 输出禁止 meta 标题
- L1 输出第一行必须是正文第一个中文段标题（即 `## 当前判断`）。
- 禁止在顶部加任何 meta 框架标题，包括但不限于：`L1 决策卡 — <case 名>` / `L1 xxx` / `L2 xxx` / `L3 xxx` / `<任意>决策卡` / `决策 card` / `轻量卡片` / `分析卡`。
- 若确需题目回显，放在 `当前判断` 首句以事实陈述方式说明（如"针对 <题目> 的判断：..."），而不是作为独立标题行。

### R-RESEARCH-001 拍板项必须先双轨研究
- 列 `请你拍板的方向` 之前，对每个候选拍板项跑**两条独立研究轨道**：
  - **轨道 A：业界 best practice** —— 先扫 `skills/` 目录看有无现成 SOP，再查 `rag/anker_values_rag_v1_zh.md` 与 `examples/`，最后看题目暗示词；必要时调用 web search。
  - **轨道 B：第一性原理** —— 回到题目本质，从公理 + 约束推出最优解（落点：经济成本结构 / 物理边界 / 数据定义层 / 用户根本需求）。
- 处理：A 与 B **高度一致** → 降到 `默认假设` 标 `[best practice + 第一性原理一致结论]`；A 与 B **分歧** → 上拍板，每个选项必须呈现两轨结论 + 在本题情境下推荐 Z 因为 ⟨综合两轨的取舍⟩。
- 禁止：(a) 没做双轨就抛给用户拍板；(b) 在轨道 A 里跳过 `skills/` 扫描；(c) 默认偏向某一轨而隐藏另一轨。

### R-CLARITY-001 拍板选项必须直接解释 + 必带引用 + union 不鼓励
- **术语必须直接解释 + 标引用来源**：含术语（"齿音"、"共振峰"、"LUFS"、"RAG 架构"等）时用"X 指 Y"句式直接说明，**并附 `[依据：…]` 或 `[S0x]` 引用**，避免编造定义。
- **选项陈述只用直接说明**：禁比喻 / 借物喻人 / "换种说法"重写（复用 R-SUM-001 到选项层）。
- **union 不鼓励但不禁止**：若 A/B/C 都成立且确有"全做"路径，可保留 D 但**必须明确标注** `D: 前 N 项叠加 = A + B + C`，并在 `我的推荐` 写出代价（成本 / 时间）。优先把"全做"作为 `你怎么回复` 里的组合方式提示。
- **元信息禁污染选项陈述**：`工作量 ×3` / `成本太高` 等只供模型评估的元信息挪到 `我的推荐` / `理由` 块。

### R-FMT-002 假定分支语法
- `假定分支` 段每条必须用 `假定情况 N（<触发条件描述>）：<对应方案动作>` 句式。
- 禁止：使用 `if ... then ...` 触发器语法。
- 原因：考试场景下用户在"思考分支"而非"执行触发器"。`假定情况 N` 让分歧点清晰，更容易快速选定一条推进。
- 合格：`- 假定情况 1（部件问题为安全/合规风险）：立刻启动延期 + 替代供应商认证 + 透明沟通`
- 不合格：`- if 部件为安全风险 then 启动延期`

### R-EVID-001 证据引用必须带锚点
- 每条 `[Sxx]` 引用必须显式指出在支撑哪个段的哪个论点，格式：
  `[Sxx]（支撑 <段名> 中 "<具体论点>"）：<引用内容>`
- 禁止：漂浮引用，如 `[S01] Anker 价值观主线：用户第一 > 短期业绩。`（没说在支撑什么）
- 原因：审阅者需要立即知道这条证据为什么出现、它在保护哪一个判断。

### R-CHECK-001 规则自检（内部执行 + 默认隐藏）
- 每次输出前，模型在内部按 10 条逐项检查（R-ASK-001 / R-ASK-002 / R-LANG-001 / R-FMT-001 / R-FMT-002 / R-RESEARCH-001 / R-CLARITY-001 / R-EVID-001 / R-SUM-001 / R-CODE-001）；任一不过，重写后再输出。
- 强制门禁：`scripts/smoke_check.sh` 在产物存盘后做结构化校验，是真实的把关层。
- **对外输出：默认不输出**这 8 条 Yes/No 列表。
- 例外：仅当用户在当前消息里显式说出 `带规则自检` / `规则自检` / `debug` / `自检模式` 之一时，才在末尾附 8 条 Yes/No。

## 六步引擎（顺序不可跳步，仅 L3 全量呈现）

L1 默认输出**不展示** STEP-1..STEP-6 章节，但 ACVD 在内部必须完整执行。仅当 Phase 3 收敛交付时按本节六步全量呈现。

### STEP-1 第一性原理拆解（ACV）

显性列举三段：

#### Axioms（公理）
不可争辩的事实，每条必须给出**可验证依据**（数据 / 定义 / 物理或经济规律 / 引用）。
- 格式：`- <事实陈述>（依据：<来源或推导>）`
- DoD：≥ 2 条；每条依据可被第三方复核。
- 禁止：把行业惯例、类比、"大家都这么做"伪装成公理。

#### Constraints（硬约束）
外部给定、本次不可突破的边界（预算、时间、合规、伦理、技术栈锁定等）。
- 格式：`- <约束名>: <数值或定性边界>（来源：<给定方>）`
- DoD：≥ 2 条；每条有明确来源方或文件出处。

#### Variables（可调变量）
可选择与权衡的维度；子问题从此派生。
- 格式：`- <变量名>（量纲/单位/取值范围）`
- DoD：≥ 2 条；每条有可量化的量纲或枚举值。

### STEP-2 问题拆解
- 输出：3–5 个子问题，每个子问题对应 STEP-1 中的一个或多个 Variables。
- 每个子问题必须标注**关键假设**及其**可验证方式**。
- DoD：子问题完整覆盖 Variables；关键假设可验证。

### STEP-3 决策账本（Decision Ledger）

给出主方案 + 备选方案。主方案的关键决策必须写入 **Decision Ledger**。

#### Decision 条目格式（4 字段齐全）

每条决策用列表条目承载，条目内以 `key: value` 四行显性写出：

```
- D<编号> <决策名>
  owner: <负责人 / 团队>
  threshold: <可数值化的触发阈值>
  trigger: <if ... then ...>（条件→动作；此处保留 if/then，因为是 Decision Ledger 内部执行触发，不是 L1 假定分支）
  rollback: <回滚条件 + 回滚路径>
```

- DoD：主方案 ≥ 3 条决策；每条 4 字段齐全；`threshold` 必须可数值化；`rollback` 必须写出回滚条件与回滚路径，不允许占位文本（如"视情况而定"）。
- 备选方案：至少 1 条，含切换条件与预期收益损益。
- 注：Decision Ledger 内部 `trigger` 字段保留 `if ... then ...` 是合规的（这是给执行者读的运行时触发），R-FMT-002 仅约束 L1 `假定分支` 段。

### STEP-4 审核优化
- 输出：至少 3 类风险，每类含：
  - 风险点 → 监控指标 → 纠偏动作
- DoD：风险、监控、纠偏三项齐全。

### STEP-5 产物呈现（.md 为主，HTML 可选）

`.md` 是**主交付产物**，直接面向审阅者阅读。必须包含：

1. **Decision Ledger 表**：同步 STEP-3 中每条 decision 的 4 字段，作为可扫读总览。
2. **KPI 看板表**：8 字段齐全 —— `metric_name / baseline / target / threshold / owner / update_freq / risk_signal / action_if_triggered`。`threshold` 必须含触发逻辑（如 `< 50% at Week 4`），不只展示静态数字。
3. **Risk 表**：同步 STEP-4 的风险 → 监控 → 纠偏三列。

HTML 何时配套：题目明确要求可视化对比 / 用户显式要 / 题目涉及波形/频谱/流程图等。结构参考 `examples/golden_case.html`。

- DoD（.md）：三张表齐全；KPI 8 字段；阈值含触发逻辑。

### STEP-6 最终方案
- 输出：完整 `.md` 交付结构。
- 含：**Executive Summary**（3–5 句，先结论后依据）+ 主判定 + 备选方案 + 关键风险 + `[S01]` 引用（带 R-EVID-001 锚点格式）。
- 禁止：口播、演讲稿语气、"接下来我们讲"式措辞。
- DoD：形成完整可审阅文档。

## 证据引用规则
