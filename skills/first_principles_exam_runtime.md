# Skill: First-Principles Exam Runtime

> 目的：把 Tesla/Elon 式第一性原理做成**可执行 SOP**。任何 case 复用此 skill，可稳定输出 `Axioms → Variables → Decisions → Rollback` 闭环。
>
> 触发条件：CLAUDE.md 的六步引擎进入 STEP-1 时强制加载；或用户显式调用 `/first_principles`。

---

## 核心判据：什么是 axiom、什么不是

一条陈述是 axiom，当且仅当**同时**满足以下 3 条：

1. **可被实证验证**：有数据、有可复现测量，或可被第三方在 24h 内核查。
2. **可被一阶原理推导**：从物理、经济成本结构、合规底线、数据定义层直接推出，不需要依赖"行业经验"。
3. **去掉它结论即崩**：若该陈述不成立，本 case 的整个决策空间会被推翻。

不满足以上任一条的，是**假设**（assumption）或**惯例**（convention），不是 axiom，应降级到子问题的"关键假设"列表里。

## SOP 三步：从 case 题到 ACVD

### 步骤 1 — 剥除类比与惯性

扫描你对 case 的第一反应表述，找出并剔除以下三类污染：

| 污染类型 | 识别信号 | 处理 |
|----------|----------|------|
| 类比型假设 | "就像 X 行业那样" / "参考 Y 公司的做法" | 删掉；追问：X/Y 成立的底层条件此处是否成立 |
| 惯性型假设 | "通常 / 一般 / 行业常规" | 删掉；追问：这个"通常"背后的数字是多少，来源是什么 |
| 权威型假设 | "某某专家说" / "报告里写的" | 保留结论但**降级为假设**；找一阶依据再升级为 axiom |

执行产物：一份**被剔除的伪公理清单**（内部草稿，不入最终交付，但便于自检）。

### 步骤 2 — 还原到公理层

对剩余陈述逐条追问："这句话若要为真，最底层需要什么成立？"问到不能再问为止。常见落点：

- **经济层**：单位成本结构（人力/算力/物料的元/件/次）、边际成本、规模曲线。
- **物理层**：时延、带宽、吞吐、能耗的物理上限。
- **合规层**：法规、合同、数据保护的硬边界。
- **数据定义层**：指标如何计算、样本如何抽取、口径差异在哪。

每条落点写成 axiom 格式：`<事实陈述>（依据：<数据/规律/引用>）`。

### 步骤 3 — 从公理重构决策空间

把 axioms + constraints 作为输入，**不看既有方案**，枚举可调的 variables。常见维度：

- 覆盖范围（哪些用户 / 场景 / 阶段）
- 强度（自动化比例 / 置信度阈值 / 采样率）
- 节奏（上线周期 / 扩展步长 / 回滚窗口）
- 资源分配（预算切分 / 人员分工 / 技术栈选择）

每条 variable 写出量纲或取值范围。这一步的输出 = STEP-2 子问题的来源。

## Decision Ledger 模板

每条决策必须 4 字段齐全。示例对比：

### 好的（通过）

```markdown
- D1 AI 覆盖率分层上线
  owner: AI 产品负责人（张三）
  threshold: Phase 1 覆盖工单类型 Top 5（约 40% 工单量）；Week 4 AI 解决率 ≥ 50%
  trigger: if Week 4 AI 解决率 < 50% then 冻结 Phase 2 扩展，回到 Phase 1 调参
  rollback: if Week 6 AI 解决率仍 < 50% then 切回全人工，并启动"外包 + AI 辅助"备选；回滚路径：关闭 AI 网关 → 流量转人工坐席 → 48h 内扩 HR 5 FTE
```

### 坏的（会被门禁拒）

```markdown
- D1 上线 AI 客服
  owner: 待定
  threshold: 视情况而定
  trigger: 如果效果不好就调整
  rollback: 必要时回滚
```

坏例失败原因：`owner` 占位、`threshold` 不可数值化、`trigger` 无 if/then、`rollback` 无条件无路径。

## Failure Fallback：数据缺失时怎么办

当 axiom 所需数据在 case 条件中未给出时，**不允许编造数字**。采用以下处理：

1. **声明缺失**：在 axiom 条目里写 `<陈述>（依据：数据缺失，使用代理指标 <X>；待补：<Y 数据源>）`。
2. **代理指标**：选用间接但可采集的指标临时替代（如缺"人均单件成本"时用"客服团队月薪总额 / 月工单量"）。
3. **返修承诺**：在 STEP-6 Executive Summary 末尾标注"本方案依赖的 <数据名> 需在 <时间> 前补齐校准"。
4. **止损预设**：在对应 decision 的 `rollback` 里写明"若代理指标误差 > 20% 则触发重算"。

## 质量自检 6 条（case 输出前必过）

1. **Axiom 可验证**：每条 axiom 的依据是否可被第三方 24h 内核查？
2. **Constraint 有来源**：每条 constraint 是否标注了给定方或文件出处？
3. **Variable 有量纲**：每条 variable 是否给出数值单位或枚举集？
4. **Decision 有 trigger**：每条 decision 的 `trigger` 字段是否是 `if ... then ...` 句式？
5. **Rollback 有路径**：每条 decision 的 `rollback` 是否写出**条件**与**回滚的具体操作步骤**，不是"必要时回滚"？
6. **[S01] 存在**：答案中是否包含至少一处 `[S01]` 引用？

6 条任一不过 → 不得输出，回到 SOP 对应步骤修复。

## 渐进披露模式（默认）

ACVD 三段**在内部必须完整执行**（Axioms → Constraints → Variables → Decisions），但**对外默认只暴露 L1**。L3 全量结构只在用户显式请求 `完整版 / 给我全量 / L3 / full report` 时才展开。

### 推进优先于提问

缺数据时**必须**先走上文「Failure Fallback」：代理指标 + 标注假设继续推进。**不允许**停下来等用户答数据。

### 问题分级（与 `CLAUDE.md` 对齐）

| 级别 | 定义 | 处理 |
|------|------|------|
| P0 | 不答即结果必错的前置条件 | 进 L1 `请你拍板的方向`，立刻提，不攒 |
| P1 | 影响主方案选型但不阻塞推进 | 进 L1 `请你拍板的方向`；超过 3 条时进 `examples/question_ledger_template.md` 排队 |
| P2 | 可用合理默认填补 | 进 L1 `默认假设`，标 `⚠ assumed` + 回退预案，**不进** `请你拍板的方向` |

一次对外 `请你拍板的方向` 条数 `<= 3`（R-ASK-002）。

注：`请你拍板的方向` 的范围比"补数据"宽得多，包括 (a) 应用第一性原理时的切入角度选择、(b) 方法论选择（jobs-to-be-done vs 用户研究 vs 数据归因）、(c) 做完 research 后的方案 commitment。这些是模型不能替代的、必须用户 in-the-loop 选择的协作介入点。

## 决策点双轨研究 SOP（R-RESEARCH-001 配套）

每个候选拍板项在写进 L1 之前，模型在内部跑**两条独立轨道**，再综合评审。

### 轨道 A：业界 best practice

按成本由低到高检索：

1. **本仓库 `skills/` 目录**（必查，最低成本最高价值）：扫一遍现有 skill 是否覆盖（HTML 制作、可视化范式、常见算法/库选型、SOP 等）。**找到可复用 SOP 时直接采用**。
2. **本仓库 `rag/anker_values_rag_v1_zh.md`**：是否有价值观或案例指引。
3. **本仓库 `examples/`**：golden_case / lite_case 是否已有同类拆解可参照。
4. **题目原文**：扫暗示词（"教学" / "评估" / "对比" / "科普" 等），这通常已锁定 best practice 适用方向。
5. **Web search**（必要时）：成熟领域拉外部 best practice 引用，标准 / 论文 / 行业白皮书优先于博客。

产物：一句话"业界做法是 X，依据 [来源]"。

### 轨道 B：第一性原理切入

回到题目本质问：**若不参考任何既有做法，从公理 + 约束推出来的最优解是什么**？常见落点：

- **经济成本结构**：单位成本 / 边际成本 / 规模曲线
- **物理可行边界**：时延 / 带宽 / 吞吐 / 能耗上限
- **数据定义层**：指标如何计算 / 样本如何抽取 / 口径差异
- **用户根本需求**：剥掉 feature 包装后的核心 job-to-be-done

产物：一句话"第一性原理推导出 Y，依据 [公理/约束]"。

### 轨道 C：综合评审

把 A 与 B 摆在一起：

| A vs B 关系 | 处理 |
|-----------|------|
| **高度一致 + 无歧义** | 降到 `默认假设`，标 `[best practice + 第一性原理一致结论]`，不上拍板 |
| **方向相同但参数不同** | 上 `请你拍板的方向`，选项里呈现两种参数取值 |
| **方向分歧** | 上 `请你拍板的方向`，每个选项必须呈现"业界做法 X / 第一性原理 Y"两轨结论 + 在本题情境下推荐 Z 因为 ⟨综合两轨的取舍⟩ |
| **A 缺失（新领域，无 best practice）** | 仅以 B 为依据上拍板，注明"业界尚无成熟 best practice" |
| **B 缺失（公理不完整）** | 仅以 A 为依据上拍板，注明"第一性原理推导依赖未给定的数据" |

### 写选项时的纪律

- **术语必须直接解释 + 标引用**：含术语时用"X 指 Y"句式，附 `[依据：…]` 或 `[S0x]`，避免编造（R-CLARITY-001）
- **直接说明，禁比喻**：选项陈述用直接事实表述，不用比喻 / 借物喻人 / "换种说法"重写（复用 R-SUM-001 到选项层）
- **推荐项必须有双轨依据**：在 `理由` 块写"业界做法 X / 第一性原理 Y / 综合本题选 Z"，不只写个人偏好
- **union 不鼓励但不禁止**（R-CLARITY-001）：若 A/B/C 都成立且确有"全做"路径，可保留 D 但必须标 `D: 前 N 项叠加 = A + B + C`，并在推荐里写出代价（成本 / 时间）
- **元信息禁污染选项陈述**："工作量 ×3" / "成本太高" 类挪到 `理由` 块，不出现在选项标题

### 反例对照

❌ 不合格（违反 R-RESEARCH-001：缺第一性原理轨道）：
```
- 音频素材来源 ｜ 选项: A 预置音频 / B 用户上传 / C 静态 PNG ｜ 我建议: A
```
问题：缺双轨研究痕迹，且业界 + 第一性原理一致（单文件零依赖最低传递摩擦），应降到 `默认假设`。

✅ 合格（两轨一致 → 降级到默认假设）：
```
默认假设：
- 音频素材采用 2-3 段预置示例 + base64 内嵌
  [best practice + 第一性原理一致结论：单文件 HTML 教学页业界标准 + 零依赖最低传递摩擦]
  ⚠ assumed（回退：若需用户上传 mp3/wav 现场处理，切 WebAudio + Meyda.js 方案）
```

❌ 不合格（违反 R-CLARITY-001：术语无解释无引用）：
```
- C. 真伪指纹页
```
问题：用户不知道"真伪指纹"是什么；模型也可能在内部编造定义。

✅ 合格（术语直接解释 + 带引用）：
```
- 决策点 1: 目标场景（这个页面服务什么目的？）

  选项:
    A. 教学科普页 — 给非专业读者直接讲"齿音是什么"，配前后音频对比
    B. 内部质检看板 — 给团队看预处理是否达标，含 LUFS 响度 / 底噪 / 失真等数值指标 [依据：EBU R128 响度标准]
    C. 真伪指纹页 — "真伪指纹"指通过共振峰位置 / 基频抖动 / 频谱滚降等声学特征判断音频是真人录制还是 AI 合成 [依据：Praat 语音分析手册 §6.2]

  我的推荐: A
  理由:
    - 业界做法：交互式音频教学页（如 Ableton Learning Synths / SoundGym）的标准结构是"前后对比 + 一句话解释"，A 直接对齐。
    - 第一性原理：题目首句问"什么是齿音"，受众是入门用户，最低认知摩擦的展示形态就是"听到差异 + 一句话定义"，与 A 同向。
    - 综合：A 在两轨上都 dominant，B/C 都需要预设专业知识。

  你怎么回复:
    - 说"走 A" 推进教学科普页
    - 说"走 B" / "走 C" 换方向
    - 说"组合 A+B" 让我合并教学 + 质检（多 Tab 形式）

- 决策点 2: ...（同结构；最多 3 条）
```

### L1 常见错误（必拒）

以下形态一律重写，不得发送：

**错误 1：meta 标题包装（违反 R-FMT-001）**

```
L1 决策卡 — 上市前供应链部件问题

## 当前判断
...
```

正确做法：删掉首行 meta 标题，直接从 `## 当前判断` 起笔。若需回显题目，写进 `当前判断` 段首句。

**错误 2：纯英文标签（违反 R-LANG-001）**

```
## Now
...

## Branches
...
```

正确做法：替换为 `## 当前判断 / ## 假定分支 / ## 请你拍板的方向 / ## 默认假设 / ## 证据 / ## 你的下一步`。

**错误 3：中英混写（违反 R-LANG-001）**

```
## 请你拍板的方向（Decisions）
```

正确做法：括号里的英文注释也要删，只保留 `## 请你拍板的方向`。

**错误 4：假定分支用 if-then 触发器语法（违反 R-FMT-002）**

```
## 假定分支
- if 部件为安全风险 then 启动延期
- if 部件为性能问题 then 小批量首发
```

正确做法：

```
## 假定分支
- 假定情况 1（部件为安全/合规风险）：立刻启动延期 + 替代供应商认证 + 透明沟通
- 假定情况 2（部件为性能/良率问题）：小批量首发 + 双供应商并行爬坡
```

**错误 5：证据漂浮无锚点（违反 R-EVID-001）**

```
## 证据
[S01] Anker 价值观主线：用户第一 > 短期业绩。
```

正确做法：

```
## 证据
- [S01]（支撑 当前判断 中 "用户第一优先于发布时间"）：Anker 价值观主线 —— 长期主义优先于抢时间窗口。
```

**错误 6：末尾输出规则自检 8 条 Yes/No（违反 R-CHECK-001 默认隐藏）**

```
## 规则自检
- R-ASK-001: Yes
- R-ASK-002: Yes
...
```

正确做法：默认不输出，内部执行即可。仅当用户当前消息含 `带规则自检` / `debug` 时才输出。

**错误 7：你的下一步段写空话**

```
## 你的下一步
请告诉我您的反馈，我将进一步优化方案。
```

正确做法：

```
## 你的下一步
1. 选一个假定分支推进（比如说"走假定情况 2"）
2. 或回答"请你拍板的方向"中的任意一条决策点
3. 或补充关键数据：当前竞品上市时间 + 替代供应商已知备选清单
```

### L1 输出自检 10 条

L1 发送前必过：

1. 第一行是 `## 当前判断`，**无** `L1 决策卡` / `L1 xxx` / `决策卡` 等 meta 标题。
2. 所有段标题全中文新 6 段名（无 `下一步动作` / `不确定的问题` 旧名残留；无 `Now` / `Next` / `Blockers` 等英文）。
3. `当前判断` 是一句话，单主语单动词，≤ 3 行。
4. `假定分支` 每条用 `假定情况 N（<条件>）：<动作>` 句式，**无 if/then 残留**（R-FMT-002）。
5. `请你拍板的方向` 条数 `<= 3`；每条已先做**双轨研究**（R-RESEARCH-001：A best practice + B 第一性原理），两轨一致的已降到 `默认假设`。
6. 每条拍板项是**多行展开**格式（含 `选项:` + `我的推荐:` + `理由:`（含 A/B/综合三行）+ `你怎么回复:` 四块），**禁单行 `｜` 拼接**。
7. 每个选项含术语时已用"X 指 Y"句式直接解释 + 标 `[依据：…]` 或 `[S0x]` 引用；选项陈述无比喻 / 无借物喻人；union 选项若出现已标 `D: 前 N 项叠加`（R-CLARITY-001）。
8. `[S01]` 引用存在，且带 `（支撑 <段名> 中 "<论点>"）` 锚点（R-EVID-001）。
9. `你的下一步` 段存在，含 2-3 行明确动作（非空话）。
10. **末尾不输出**规则自检 10 条 Yes/No（除非用户当前消息含 `带规则自检` / `debug` 触发）。

任一不过 → 重新裁剪、降级到 `默认假设`、补 gloss、再输出。
